WorkIndia — Performance Marketing Automation

3m read
Experimentation Segmentation Growth ML

🔍 Problem

Candidates were being sent job notifications at suboptimal times and with generic copy, resulting in low open rates and poor click-to-apply conversion. Budget was being wasted on poorly targeted outreach.

💡 Hypothesis

If we personalise when and to whom we send notifications — based on individual engagement history and job market signals — we will see a measurable lift in application rates without increasing overall notification volume.

🔧 Approach

  • Automated WhatsApp campaigns via cron jobs + user segmentation to send messages during each user's peak engagement window.
  • Analysed CPD (Cost per Download) and CPR (Cost per Registration) across channels (MySQL + Metabase) to reallocate budget to highest-ROI segments.
  • Studied 2024–2025 tech job market trends to surface high-demand roles and improve default targeting logic.

⚖️ Key Decisions

  • Time-affinity over broadcast: Chose to compute individual send-time slots rather than a single best time for the cohort — more engineering work, but key for a diverse user base.
  • Downstream metric tracking: Anchored success on job applications, not open rate — preventing optimization for vanity metrics.
  • Budget reallocation ≠ budget cut: Improved allocation efficiency rather than reducing spend, maintaining reach while cutting waste.

📊 Impact

+7% Engagement Efficiency
+23% Budget Optimization
+11% Targeting Relevance

🛠️ Engineered Systems

Problem: Generic broadcast notifications had poor CTR and contributed to user fatigue and unsubscribes.

System built: A cron-based segmentation engine that identifies each user's historical engagement window and schedules notifications within a personalised ±2-hour band.

Key decision: Chose a lightweight rule-based model over a heavy ML model to keep latency low and the system debuggable.

Problem: Manually writing notification copy for dozens of user segments was slow and inconsistent.

System built: An agentic pipeline that takes a segment profile + job category as input and generates personalised, on-brand copy variants for A/B testing — ready for direct injection into the campaign scheduler.

Impact: Read-rates improved by 15.6%, click-rates by 4%, and click-to-install rates by 9.2%.

Key decision: Used structured output + prompt guards to ensure copy remained within character limits and brand voice guidelines automatically.

Problem: Ad-hoc queries for notification metrics were slow, inconsistent, and hard to reproduce across teams.

System built: A star-schema data model with a notifications fact table (one row per event), dimension tables for time, channel, segment, and job category, and pre-built aggregators for L0 metrics.

Key decision: Designed for query-time flexibility — raw events in the fact table, not pre-aggregated — so new metric definitions don't require schema changes.

🚀 What I'd Do Next

  • Build a real-time feedback loop: if a user ignores 3 consecutive notifications, suppress for 5 days and re-enter with a completely different job category.
  • Run an A/B test on notification copy: data-driven subject lines vs. benefit-led copy.
  • Explore multi-arm bandit allocation for budget across channels — move from rule-based to adaptive allocation.

Experimentation & Growth Systems

Building infrastructure that makes decisions repeatable, not one-off.

🧪
A/B Testing via Cron Segmentation

Segmented user cohorts by engagement history to run controlled timing experiments — moving from intuition-based send times to data-validated windows.

  1. Segment users into buckets: Morning (6–10am), Afternoon (12–3pm), Evening (6–9pm) based on 30-day historical opens.
  2. Hold 10% of each bucket as a control group receiving current default send time.
  3. Measure CTR and downstream application rate (not just opens) as the primary success metric.
  4. Promote winning window per segment after 14-day observation cycle.
📐
Metric Design (Weighted Metrics)

Designed composite notification health metrics that balance engagement with user fatigue — avoiding optimisation for a single vanity metric.

Notification Health Score =

(0.4 × Click Rate) + (0.4 × Application Rate) − (0.2 × Unsubscribe Rate)

This formulation ensures that campaigns can't win by sacrificing long-term user trust for short-term clicks.

🔄
Auto-Refresh System (14-day cycle)

Built a pipeline that automatically re-evaluates user segment assignments every 14 days based on fresh engagement data — preventing segment staleness.

  1. Cron job runs every 14 days post-campaign cycle.
  2. Re-computes time affinity scores from last 30 days of event data.
  3. Users with drift > 20% in engagement window are re-bucketed.
  4. All changes logged for retrospective analysis and audit trail.

How I Think About Products

Mental models I use to make better product decisions.

01
Optimise for downstream metrics, not vanity metrics

Open rate is a proxy. Application rate is a business outcome. I always trace the metric chain to find what actually matters and set guardrails on the rest.

02
Balance engagement with user fatigue

More notifications ≠ more engagement. Every product decision has an attention budget. I always model the unsubscribe cost alongside the click gain before shipping a frequency change.

03
Prefer systems over one-off solutions

A script that runs once is technical debt. A pipeline that runs every 14 days, logs itself, and handles edge cases is a product. I build for the second run, not the first.

04
Use data to guide, not dictate decisions

Data shows what happened; it rarely explains why. I combine quantitative signals with user behaviour intuition and business context — never outsourcing the decision to a number alone.

05
Ship experiments, not assumptions

Every product strong opinion is a hypothesis in disguise. I default to small, fast experiments before committing to large builds — minimising the cost of being wrong.

06
Make the invisible legible

The best product insight is the one nobody sees yet — buried in event logs, latency spikes, or drop-off points. I treat data exploration as a product skill, not just an analytical one.
